## Codebase Patterns
- Backstage catalog model and validators live in `lib/catalog-shared.ts`; entity refs normalize to `Component:default/<name>` when kind/namespace are omitted and enforce Backstage kind casing.
- Relationship derivation and entity enrichment live in `lib/catalog-core.ts` (no fs/path deps — safe for client bundles and Vitest isolation). Core exports: `deriveCatalogRelationships`, `attachCatalogRelationships`, `filterCatalogEntities`, `groupCatalogEntities`, `getCatalogFacets`.
- `lib/content.ts` is the SSR entry point: imports from catalog-core and wraps file I/O (yaml loader, markdown reader).
- `normalizeTarget(raw, fallbackKind)` in catalog-core handles `namespace/name` (without kind prefix) by prepending the fallback kind — needed because Backstage YAML often omits the kind prefix for same-kind refs.
- Tests run with Vitest via `npm test`; vitest.config.ts uses `@vitejs/plugin-react` for JSX transforms (Next.js sets jsx:preserve which breaks plain esbuild).
- Fixture YAML files for integration tests live under `tests/fixtures/catalog-accessors/catalog-info.yaml` and `content/catalog/catalog-info.yaml`.
- `dependsOn` references without a kind prefix fall back to `Component`; use explicit refs like `Resource:default/name` to target other kinds.
